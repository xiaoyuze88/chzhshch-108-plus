# 缠论知识库 (chzhshch-108-plus)

将缠中说禅 108 课博客原文，经 AI 整理提炼为**优先面向 AI agent 消费的结构化知识体系**。

## 设计思想

本知识库的整理方式优先考虑 AI 消费效率——让 AI 在有限上下文窗口内，以最低 token 成本获取最精确、最完整的缠论知识。人类当然也可以读，但行文风格和组织方式服务于机器可判定性，而非阅读流畅性。

> 内容为纯 AI 提炼、AI复核，未经人类校对，不能100%保证准确性和完整性。

**为此做出的设计选择**：

- **充要条件而非描述**：每个定义/定理都写成可形式化判定的充要条件，而非自然语言描述——AI 不需要"理解"，只需要"匹配"
- **附录速查层**：将定义、定理、操作规则从正文中抽出，建立 A~E 五个速查附录，AI 可以先做低成本命中检测，避免全量阅读
- **知识图谱导航**：graphify 构建的概念图谱让 AI 通过 BFS 定位相关文件，而非逐篇扫描
- **前置知识显式化**：每个文件头标注依赖关系，AI 可以按需递归加载，不会遗漏上下文
- **判断树/决策表**：操作类内容写成判断树和决策表，AI 可以直接沿分支执行，无需推理

**对比传统整理方式的差异**：

| | 传统教材 | 本知识库 |
|---|---------|---------|
| 读者 | 人类为主 | AI agent 优先 |
| 定义风格 | 描述性，需理解 | 充要条件，可判定 |
| 查询路径 | 从头读到尾 | 附录命中→图谱定位→正文补充 |
| 操作指引 | 原则性描述 | 判断树+触发条件+冲突优先级 |
| 溯源 | 无 | 每条标注课文编号，可回溯原文 |

## 依赖：[graphify](https://github.com/safishamsi/graphify)

将文本文件转化为可查询的知识图谱。`pip install graphifyy` 安装，`/graphify <path>` 构建，`/graphify query "关键词"` 查询。详见 [graphify 仓库](https://github.com/safishamsi/graphify)。

本项目维护三套图谱：

| 图谱目录 | 数据来源 | 用途 |
|----------|----------|------|
| `graphify-out/` | `organized-v2/` | 主图谱，日常查询用（150节点·260边·16社区） |
| `archived/graphify-out-108/` | `archived/108/` | 原文图谱，回溯博客措辞/图例/课文上下文时用 |
| `archived/graphify-out-v1/` | `archived/organized-v1/` | v1 图谱，对比 v1→v2 演化时用 |

三者对应内容的演进关系：

```
108  →  v1  →  v2
原文    聚合    体系化

archived/108/              缠中说禅原始博客，零散的108课，按时间排列
archived/organized-v1/     第一版整理，按主题聚合原文，但仍是原文拼合
organized-v2/              第二版整理，教科书式知识体系：充要条件、定理链、判断树、操作规则
```

- **108 → v1**：从零散博客按主题聚合，保留原文措辞
- **v1 → v2**：从原文拼合提炼为结构化知识体系，补充充要条件、边界判断、操作规则
- 需要溯源时，可沿 v2 → v1 → 108 逐层回溯至原始博客

常用操作：

```bash
/graphify organized-v2 --update    # 增量更新（修改 organized-v2 后执行）
/graphify query "背驰 区间套"       # BFS 查询，从关键词定位相关文件
/graphify path "走势中枢" "第三类买点"  # 两个概念间的最短路径
```

graphify 在查询流程中承担"广"的角色——当附录无法覆盖时，通过图谱定位相关文件，再回到正文阅读。

## 查询方法

### 在 AI Agent 中查询

项目 `CODEBUDDY.md` 定义了完整的查询规范，agent 会自动执行以下流程：

```
问题 → 分类路由 → 附录低成本命中检测 → 图谱定位 → 正文阅读 → 组织回答
```

核心原则：**附录负责"准"，图谱负责"广"，正文负责"深"**

- **定义类/定理类/操作类/溯源类**：先查对应附录（A~E），命中后读附录正文，必要时补正文
- **关系类/综合类/模糊提问**：先用图谱定位范围，再回附录+正文

## 搭配 Claw 类 Agent 使用

将以下提示词添加到你的 `SOUL.md`，即可让 Claw 类 Agent 在对话中自动识别并查询本知识库：

```markdown
## 缠论知识库

我有一个缠论（缠中说禅）知识库项目，位于 `<你的本地路径>/chzhshch-108-plus/`。
知识库包含体系化教材（organized-v2）、知识图谱（graphify-out）、附录速查、原始博客（archived/108）等层次。

**核心规则：当用户明确提及"缠论知识库"时，必须先完整读取 `<你的本地路径>/chzhshch-108-plus/CODEBUDDY.md` 中的"缠论知识库查询规则"部分，再按规范执行查询。**

触发词：
- "缠论知识库"、"从缠论知识库查"、"知识库里查缠论"、"缠论知识库怎么说"，或用户明确指示从本地缠论知识库中检索信息

**不触发**：用户只提到缠论概念（如"背驰"、"中枢"等）但未提及"知识库"时，正常回答或搜索，不走此流程。

**注意**：
- 缠论知识库的查询流程较复杂（问题分类→附录探测→图谱定位→正文阅读），不要跳步
- 项目 CODEBUDDY.md 中的 `graph.json` 是机器格式，只能通过 Python 脚本查询，不要直接读取
- 附录速查路径：`organized-v2/附录/A~E`，按问题类型选附录，不要横向扫描
```

使用前将 `<你的本地路径>` 替换为项目在你机器上的实际路径。
